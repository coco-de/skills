# A/B Testing (A/B 테스트)

## 트리거
- 기능 변경이나 UI 변경의 효과를 측정해야 할 때
- "A/B 테스트", "실험", "ab test", "split test" 키워드가 언급될 때
- 두 가지 이상의 대안 중 데이터 기반으로 선택해야 할 때
- 기능 롤아웃의 영향을 정량적으로 평가해야 할 때

## 동작
1. **실험 설계**:
   - 가설 수립: "[변경]은 [메트릭]을 [방향]으로 [크기]만큼 변화시킬 것이다"
   - 주요 메트릭 (Primary Metric) 및 가드레일 메트릭 정의
   - 실험군/대조군 분리 전략 설계
   - 무작위 배정 단위 결정 (사용자/세션/디바이스)
2. **샘플 크기 및 기간 산정**:
   - 최소 감지 효과 (MDE) 설정
   - 통계적 검정력 (Power, 보통 80%) 기준
   - 유의 수준 (Alpha, 보통 5%) 설정
   - 필요 샘플 크기 계산
   - 예상 트래픽 기반 실험 기간 산출
3. **실험 실행 가이드**:
   - 피처 플래그 구현 가이드
   - 이벤트 로깅 체크리스트
   - 실험 시작 전 AA 테스트 (샘플 비율 검증)
   - 모니터링 포인트 정의
4. **결과 분석**:
   - 통계적 유의성 검증 (p-value, 신뢰구간)
   - 효과 크기 (Effect Size) 계산
   - 세그먼트별 결과 분석 (이질적 처리 효과)
   - 노벨티 효과/프라이머시 효과 점검
   - SRM (Sample Ratio Mismatch) 검증
5. **의사결정 및 문서화**:
   - Ship/Don't Ship/Iterate 결정 프레임워크
   - 실험 결과 문서화 (재현 가능한 형태)
   - 학습 사항 팀 공유 템플릿

## 출력
- 실험 설계서 (가설, 메트릭, 샘플 크기, 기간)
- 샘플 크기 계산 결과
- 결과 분석 보고서 (통계적 유의성, 효과 크기, 신뢰구간)
- 의사결정 기록 (Decision Log)
- 실험 학습 요약

## 참고
- Flutter 앱에서는 Firebase Remote Config 또는 자체 피처 플래그로 실험군 분리 가능
- Jaspr Web에서는 서버 사이드 렌더링 특성상 서버에서 실험군 배정 권장
- Serverpod 엔드포인트에서 실험 배정 로직 및 이벤트 로깅 구현
- 모바일과 웹의 트래픽 차이로 인해 플랫폼별 별도 실험 또는 통합 실험 설계 고려
