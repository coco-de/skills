# Experiment Design (실험 설계)

## 트리거
- 새로운 기능이나 접근 방식의 효과를 검증해야 할 때
- "실험 설계", "experiment", "테스트 설계" 키워드가 언급될 때
- 가설 기반 의사결정이 필요할 때
- MVP나 프로토타입 테스트를 계획할 때

## 동작
1. **가설 수립**:
   - "우리는 [행동]을 하면 [결과]가 나올 것이라 믿는다" 형식으로 구조화
   - 측정 가능한 성공 기준 정의
   - 가설의 전제 조건 명시
2. **실험 유형 선택**:
   - Painted Door Test (기능 관심도 측정)
   - Wizard of Oz (수동 백엔드로 프론트엔드 테스트)
   - Concierge Test (수동 서비스 제공)
   - Fake Door Test (랜딩 페이지 테스트)
   - Prototype Test (인터랙티브 프로토타입)
3. **실험 설계서 작성**:
   - 목표 및 가설
   - 대상 사용자 세그먼트
   - 실험 기간 및 샘플 크기
   - 측정 메트릭 (주요/보조)
   - 성공/실패 기준 (사전 정의)
4. **실행 계획 수립**: 타임라인, 담당자, 필요 리소스 정의
5. **결과 분석 프레임워크 제공**:
   - 정량적 결과 해석 가이드
   - 정성적 피드백 종합 방법
   - 의사결정 매트릭스 (진행/피벗/중단)

## 출력
- 가설 문서 (구조화된 형식)
- 실험 설계서 (Markdown)
- 실험 실행 체크리스트
- 결과 분석 템플릿
- 의사결정 기록 (Decision Log)

## 참고
- Flutter 앱에서는 Firebase Remote Config 등을 활용한 피처 플래그 실험 가능
- Jaspr Web에서는 랜딩 페이지 기반 Painted Door 테스트에 적합
- Serverpod의 엔드포인트를 활용하여 서버 사이드 실험 로직 구현 가능
- BMAD의 기능 명세와 별개로, 발견 단계의 학습 목적 실험에 집중
